# 복구 가능한 에러(Result<T, E>)와 복구 불가능한 에러(panic!)

- **요약 목록**
    - **복구 불가능한 에러 (`panic!`)**: 잘못된 인덱스 접근 등 프로그램이 더 이상 실행될 수 없는 치명적인 상황에 발생하며, 프로그램을 즉시 종료하고 메모리를 정리합니다.
    - **복구 가능한 에러 (`Result<T, E>`)**: 파일 미존재나 네트워크 타임아웃 등 예측 가능하고 대처 가능한 에러에 사용하며, `Ok(T)` 또는 `Err(E)`를 반환합니다.

## 복구 불가능한 에러: `panic!`

`panic!` 매크로가 실행되면 프로그램은 에러 메시지를 출력하고, 스택을 역추적(Backtrace)하여 메모리를 정리한 뒤 **즉시 종료**됩니다.

개발자가 직접 `panic!`을 호출할 수도 있고, 배열의 범위를 벗어난 인덱스에 접근하는 등 런타임에 수습할 수 없는 치명적인 실수가 일어났을 때 Rust가 자체적으로 발생시키기도 합니다.

```rust
fn main() {
    panic!("치명적인 오류 발생! 프로그램을 종료합니다.");
}
```

## 복구 가능한 에러: `Result<T, E>`

대부분의 에러는 프로그램을 통째로 종료해야 할 만큼 치명적이지 않습니다. 예를 들어, 열려고 하는 파일이 없을 때는 프로그램을 종료하는 대신 "파일을 새로 만들거나 다른 경로를 입력받는" 식으로 복구할 수 있어야 합니다.

Rust는 이를 위해 표준 라이브러리에 다음과 같은 `Result` 열거형을 정의해 두었습니다.

```rust
enum Result<T, E> {
    Ok(T),  // 작업이 성공했을 때 반환되는 성공 데이터 (T 타입)
    Err(E), // 작업이 실패했을 때 반환되는 에러 정보 (E 타입)
}
```

### `Result` 매칭하여 처리하기

함수가 `Result`를 반환한다면, 앞서 배운 `match` 표현식을 사용하여 성공과 실패 케이스를 안전하게 갈라 처리할 수 있습니다.

```rust
use std::fs::File;

fn main() {
    let greeting_file_result = File::open("hello.txt");

    let greeting_file = match greeting_file_result {
        Ok(file) => file, // 성공 시 실제 파일 객체를 꺼냄
        Err(error) => {   // 실패 시 에러 원인에 맞게 대응
            panic!("파일을 열 수 없습니다: {:?}", error);
        }
    };
}
```

## 에러를 강제로 처리하는 지름길: `unwrap`과 `expect`

`match`를 매번 쓰는 것이 번거로울 때, `Result` 타입이 제공하는 숏컷 메서드를 사용할 수 있습니다.

### 1. `unwrap`

성공하면 `Ok` 안의 값을 반환하고, 실패하면 `panic!`을 일으켜 프로그램을 종료시킵니다. 빠르고 간단하게 프로토타이핑을 할 때 유용합니다.

```rust
let tmp_file = File::open("hello.txt").unwrap();
```

### 2. `expect`

`unwrap`과 동일하게 작동하지만, 실패하여 `panic!`이 발생할 때 **원하는 에러 메시지를 커스텀**하여 출력할 수 있어 디버깅에 훨씬 유리합니다.

```rust
let tmp_file = File::open("hello.txt")
    .expect("hello.txt 파일은 이 프로젝트에 반드시 존재해야 합니다.");
```
